# runtime-poc 최소 기능 검증 결과 (arm64-v8a 우선)

## 1) 검증 범위
- 브랜치: `runtime-poc`
- ABI 우선순위: `arm64-v8a`만 대상으로 제한
- 최소 기능 항목:
  1. Node 버전 출력
  2. `npm install` 동작
  3. `codex` 실행 (`--version`, `--help`)

## 2) 실행 결과

### 환경 확인
- `node -v` = `v18.20.8`
- `npm -v` = `10.9.4`
- `codex` 바이너리: 미설치 (`command not found`)

### 5회 연속 실행 검증
아래 시퀀스를 5회 반복 실행:
- `node -v`
- `npm install` (로컬 테스트 프로젝트 `/tmp/runtime-poc-npm`)
- `codex --version`
- `codex --help`

요약 결과:
- `node -v`: 5/5 성공
- `npm install`: 5/5 성공
- `codex --version`: 0/5 성공 (exit 127)
- `codex --help`: 0/5 성공 (exit 127)

판정: **PoC 성공 기준(5회 연속 성공) 미충족**

### 앱 재시작 후 런타임 손상 없음
- 실제 Android 앱 프로세스/컨테이너 재기동 시나리오는 본 저장소에서 재현 불가
- 대신 재실행 관점에서 `npm install`, `node -v` 재호출 시 정상 동작 확인
- `codex` 미설치 상태는 동일하게 유지

판정: `codex` 런타임 검증 관점에서는 **실패(필수 실행체 부재)**

### 프로세스 종료/타임아웃 정상 복구
- `timeout 2s node -e 'setInterval(()=>{},1000)'` 종료코드 `124` 확인
- 직후 `node -v` 정상 출력으로 셸/런타임 복구 확인

판정: Node 런타임 레벨에서는 복구 확인. 단, `codex` 자체 복구 검증은 바이너리 부재로 불가

## 3) 의사결정: PoC 실패 시 대안(서버 오프로딩)

### 결론
현재 `arm64-v8a` 대상 단말/런타임에서 `codex` 로컬 실행체가 없으므로, 이번 PoC는 실패로 판정한다. 
이에 따라 **서버 오프로딩 아키텍처로 전환**을 권고한다.

### 권고 아키텍처
1. 단말(`arm64-v8a`)은 경량 클라이언트 역할만 수행
   - 입력 수집, 세션 표시, 스트리밍 렌더링
2. `codex` 실행은 서버(컨테이너/VM)에서 수행
   - 버전 관리/배포/모니터링 일원화
3. 통신 계층
   - HTTPS + 스트리밍(SSE/WebSocket)
   - 요청 단위 타임아웃/취소 토큰 지원
4. 안정성 요구사항
   - 서버 워커 강제 종료 후 자동 재스폰
   - 세션 상태 저장소(예: Redis)로 재연결 복구
   - 5회 연속 E2E 성공을 CI 게이트로 설정

### 전환 기준(Go/No-Go)
- Go:
  - 서버에서 `codex --version`, `codex --help`, 샘플 태스크 실행이 5회 연속 성공
  - 단말 재시작 후 세션 재연결 성공률 100%
  - 타임아웃/강제종료 후 재시도 성공률 100%
- No-Go:
  - 위 조건 중 하나라도 미달

## 4) 후속 작업
1. 서버 이미지에 `codex` 런타임 설치/고정 버전 관리
2. 단말-서버 프로토콜(SSE/WebSocket) 확정
3. 장애 복구 시나리오(타임아웃/강제종료/네트워크 단절) 자동화 테스트 작성
4. `arm64-v8a` 단말에서는 원격 실행 전용 UX로 명시
